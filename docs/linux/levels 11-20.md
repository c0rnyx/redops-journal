# Level 11

## Challenge
- The password for the next level is stored in the file `data.txt`, where all lowercase (`a-z`) and uppercase (`A-Z`) letters have been **rotated by 13 positions**.

## Problem
The data is encoded with **ROT13** (a simple Caesar cipher shifting letters by 13 positions).

## Solution 
```bash
tr 'A-Za-z' 'N-ZA-Mn-za-m' < data.txt
```

## Explanation

- `tr` = **translate characters**
- `'A-Za-z'` = input character set (all uppercase and lowercase letters)
- `'N-ZA-Mn-za-m'` = output character set rotated by 13 positions

### Mapping Example

- `A → N`, `B → O`, ..., `N → A`
- `a → n`, `b → o`, ..., `n → a`

The `< data.txt` part feeds the file as input to `tr`.

This prints the decoded text directly — the **password appears in cleartext** in the output.

# Level 12

## Challenge
- The password for the next level is stored in the file data.txt, which is a hexdump of a file that has been repeatedly compressed. For this level it may be useful to create a directory under /tmp in which you can work. Use mkdir with a hard to guess directory name. Or better, use the command “mktemp -d”. Then copy the datafile using cp, and rename it using mv (read the manpages!)
    
    ### 1️⃣ Create a temporary working directory
    
    It’s best to **not work directly in the home directory**, so a safe temporary folder is created:
    
    ```bash
    tmpdir=$(mktemp -d)
    cd "$tmpdir"
    ```
    
    - `mktemp -d` → creates a directory with a **random, hard-to-guess name**.
    - `tmpdir=$(...)` → stores its path in a variable.
    - `cd "$tmpdir"` → switch to that directory.
    
    ---
    
    ### 2️⃣ Copy the `data.txt` file
    
    From the Bandit home directory:
    
    ```bash
    cp /home/bandit12/data.txt .
    ```
    
    ---
    
    ### 3️⃣ Rename the file
    
    The file is a **hexdump**, so it should be renamed to a binary file before processing:
    
    ```bash
    mv data.txt data.bin
    ```
    
    - `mv` → renames the file.
    - Now `data.bin` will be the working file.
    
    ---
    
    ### 4️⃣ Convert the hexdump back to binary
    
    Bandit hints that `data.txt` is a **hexdump**, probably generated by `xxd`. To restore the original binary:
    
    ```bash
    xxd -r data.bin > data.orig
    ```
    
    - `xxd -r` → reverses a hex dump back into binary.
    - `data.orig` → now a binary file (likely a compressed archive).
    
    ---
    
    ### 5️⃣ Decompress repeatedly
    
    Now, the tricky part: the file has been **repeatedly compressed**, often alternating formats like `gzip` and `bzip2`.
    It is therefore necessary to be decompressing it in a loop:
    
    ```bash
    file data.orig
    ```
    
    - This displays the current file type (gzip, bzip2, tar, etc.).
    
    Then the appropriate command is used:
    
    - For gzip:
        
        ```bash
        mv data.orig data.gz
        gunzip data.gz
        ```
        
    - For bzip2:
        
        ```bash
        mv data.orig data.bz2
        bunzip2 data.bz2
        ```
        
    - For tarballs:
        
        ```bash
        tar xf data.tar
        ```
        
    
    > The file and the decompress step needs to be repeated until file reports a plain text file.
    > 
    
    ---
    
    ### 6️⃣ Read the password
    
    Once the file is finally **plain text**, just:
    
    ```bash
    cat <filename>
    ```

# Level 13

## Challenge
- The password for the next level is stored in **/etc/bandit_pass/bandit14 and can only be read by user bandit14**. For this level, you don’t get the next password, but you get a private SSH key that can be used to log into the next level. **Note:** **localhost** is a hostname that refers to the machine you are working on.

## Solution
- 1️⃣ The private key needs to be copied onto the **local machine**
- 2️⃣ and saved locally, for example:
    
```bash
nano /tmp/bandit14_key
# paste the key
chmod 600 /tmp/bandit14_key
```
-> SSH from the **local machine** to the Bandit server on the **special port (2220)**:  
   ```bash
   ssh -i /tmp/bandit14_key -p 2220 bandit14@bandit.labs.overthewire.org
   ```

- The login is done as `bandit14`.
- Then the password can be read:
    
```bash
cat /etc/bandit_pass/bandit14
```

# Level 14

## Challenge

- The password for the next level can be retrieved by submitting the password of the current level to port 30000 on localhost.

## Solutiuon

  Send the password of **bandit14** to port 30000 on localhost:
  
  ```bash
  cat /etc/bandit_pass/bandit14 | nc localhost 30000
  ```
    
  - `cat /etc/bandit_pass/bandit14` → reads the **password of the current level**, which is bandit14’s password.
  - The pipe `|` → sends that password **as input** to the next command.
  - `nc localhost 30000` → connects to **TCP port 30000 on the local machine** and sends whatever is piped in (`bandit14`’s password).
  - The server listening on port 30000 receives your password, verifies it, and responds with the **next level’s password** (bandit15).

## Explanation

- Bandit 14’s level challenge is specifically set up to require **the current level password** to be sent to this service.
- Only bandit14 can read `/etc/bandit_pass/bandit14` because of Linux file permissions.
- Port 30000 is a little “mini server” running locally that expects **the current password on stdin** and responds with the next password.
    
## Troubleshooting common mistakes
    
- Trying to do this **as a different user** (bandit13, bandit15, etc.) → fails because you **can’t read the password file**.
- Forgetting the pipe → `nc` won’t receive anything to check.
- Forgetting a newline → some services require the password to end with `\n` (though usually `cat` adds it automatically).****

# Level 15

## Challenge

- The password for the next level can be retrieved by submitting the password of the current level to port 30001 on localhost using SSL/TLS encryption.
    
## Solution

  Variant A: pipe the password into openssl
 ```bash
 printf "%s\n" "$(cat /etc/bandit_pass/bandit15)" | openssl s_client -connect localhost:30001 -quiet 2>/dev/null
 ```

   or

  Variant B: redirect the password file to openssl
  ```bash
  openssl s_client -connect localhost:30001 -quiet < /etc/bandit_pass/bandit15 2>/dev/null
  ```

## Explanation
  
- `cat /etc/bandit_pass/bandit15` (or the `printf` wrapper) : reads your current-level password and ensures a trailing newline.
- `|` or `<` : sends that password to the TLS client’s stdin.
- `openssl s_client -connect localhost:30001` : opens an SSL/TLS connection to localhost port 30001.
- `quiet` : suppresses some of `s_client`’s verbose session chatter.
- `2>/dev/null` : hides remaining diagnostic output (cert info etc.), leaving you with the server response (the next password).

# Level 16

## Challenge

- The credentials for the next level can be retrieved by submitting the password of the current level to a port on localhost in the range 31000 to 32000. First find out which of these ports have a server listening on them. Then find out which of those speak SSL/TLS and which don’t. There is only 1 server that will give the next credentials, the others will simply send back to you whatever you send to it.
    
## Explanation (summary of steps)
    
1. **Discover listening ports in range 31000–32000.**
- `nmap` was used to find which ports in that range had services listening. `nmap -p 31000-32000` showed five open ports: `31046 31518 31691 31790 31960`.

```bash
 PORT STATE SERVICE 
    
  31046/tcp open unknown
    
  31518/tcp open unknown 
    
  31691/tcp open unknown 
    
  31790/tcp open unknown 
    
  31960/tcp open unknown

```
   
2. **Probe each port to find which speak TLS vs plain TCP.**
- `openssl s_client` handshake tests indicated `31518` and `31790` are TLS services (they returned certificate info), while the others behaved like plain TCP echo services.

TLS handshake probe (fast, non-blocking)
  
```bash
timeout 2 bash -c 'echo | openssl s_client -connect 127.0.0.1:PORT 2>&1' | sed -n '1,6p'
```
  
- If output contains `CONNECTED` or `----BEGIN CERTIFICATE-----` it’s TLS.
     → The output from this command showed two ports running TSL certificates. The responses were as such: 
```bash
bandit16@bandit:~$ timeout 2 bash -c 'echo | openssl s_client -connect 127.0.0.1**:31518** 2>&1' | sed -n '1,6p' Can't use SSL_get_servername depth=0 CN = SnakeOil verify error:num=18:self-signed certificate verify return:1 depth=0 CN = SnakeOil verify return:1
```

**MEANING:** 

- `31518` and `31790` — output shows a certificate (`depth=0 CN = SnakeOil`) → **these are TLS** services.
- `31046`, `31691`, `31960` — output shows `CONNECTED` and `no peer certificate available` / `unexpected message` → `openssl` connected at TCP level but the peer did not perform a TLS handshake, so these are **plain (non-TLS) TCP** services (often echo servers that speak plain text).


3. **Send the current password to each candidate over the correct protocol.**
- For TLS ports the `openssl s_client` was used and the password was piped into it; for plain TCP `nc` would be used.
- Each echo server simply returned the same password back. The correct server returned a **different** single-line response — that different string is the **next-level password**.

 ```bash
 PASS="$(cat /etc/bandit_pass/bandit16)"; for p in 31518 31790; do echo "---- $p ----"; printf "%s\n" "$PASS" | timeout 3 bash -c "openssl s_client -connect 127.0.0.1:$p -quiet 2>/dev/null" | sed -n '1,8p'; done
 ```
    
## Line-by-line explanation

```bash
PASS="$(cat /etc/bandit_pass/bandit16)"; for p in 31518 31790; do echo "---- $p ----"; printf "%s\n" "$PASS" | timeout 3 bash -c "openssl s_client -connect 127.0.0.1:$p -quiet 2>/dev/null" | sed -n '1,8p'; done
```
- `PASS="$(cat /etc/bandit_pass/bandit16)"`
  - **Reads the current-level password** from the file into the shell variable `PASS`.
  - Using `$(...)` captures the command output exactly (no extra newlines beyond what the file has). Storing it in a variable lets us reuse it easily.
    
- `;`
  - Separator so the `for` loop can be ran next on the same command line.
    
- `for p in 31518 31790; do ...; done`
  - A shell loop that runs the inner block once for each port (`p`) listed. These were the two ports that were detected earlier as TLS-capable.
    
- `echo "---- $p ----"`
  - Prints a small header so the outputs from different ports are easy to distinguish.
    
- `printf "%s\n" "$PASS"`
  - **Prints the password followed by a newline**. 
  - `printf` is used instead of `echo` because it is consistent and won’t reinterpret escape sequences or change behavior with odd characters. The `\n` ensures the remote service receives a newline (like pressing Enter).
    
- `|`
  - Pipes the printed password into the stdin of the next command. So the password becomes the input to `openssl s_client`.
    
- `timeout 3 bash -c "openssl s_client -connect 127.0.0.1:$p -quiet 2>/dev/null"`
  - `timeout 3` — ensures the whole `openssl` attempt will be killed after 3 seconds if it hangs (avoids you waiting forever).
  - `bash -c "..."` — runs the quoted command string **in a shell** (`timeout` wraps the whole `openssl` invocation reliably).
  - `openssl s_client -connect 127.0.0.1:$p -quiet` — opens an SSL/TLS connection to the local port `p`, reads from stdin (the password that was piped) and sends it over the TLS channel. `quiet` suppresses most of `openssl`'s verbose handshake text so the server reply is easier to read.
  - `2>/dev/null` — discards `openssl`'s diagnostic output (stderr), leaving only the server response on stdout (which is what we want to inspect).
    
- `| sed -n '1,8p'`
  - Pipes the server response into `sed` and prints only the **first 8 lines**. This prevents huge or noisy responses from cluttering the terminal and surfaces the likely single-line password which is usually at the top.

    
**Data flow**:
    
`password file` → `printf` → pipe → `openssl s_client` (TLS connect to server) → server responds → `sed` prints first lines.

4. **Identify the correct response.**
- The returned outputs were inspected; the one that differed from what was sent (and looked like a Bandit password/token) was taken as the bandit17 password.

# Level 17

## Challenge 

- There are 2 files in the homedirectory: passwords.old and passwords.new. The password for the next level is in passwords.new and is the only line that has been changed between passwords.old and passwords.new

## Problem

There is:

  - `passwords.old`
  - `passwords.new`
    
The password that is needed is **the only line that differs between the two files**.

---
    
## Solution
  
`diff`:
    
```bash
diff passwords.old passwords.new
```
This will show output like:
    
 ```
 < oldpasswordline
 > newpasswordline
 ```
  
The line with the **greater-than sign (`>`)** is the one from `passwords.new`.
    
-> **That is the password for bandit18.**  

---

Option B:
To show only the new line:  
  ```bash
  diff passwords.old passwords.new | grep ">"
  ```
    
  ## Explanation
- The challenge states exactly one line changed.
- `diff` compares each line in both files.
- The changed line in `passwords.new` is the new password.

# Level 18

## Challenge 
The password for the next level is stored in a file readme in the homedirectory. Unfortunately, someone has modified .bashrc to log you out when you log in with SSH.

## Problem
  Normally we log in via SSH and get a shell.
  But here, `.bashrc` in the home directory is set up to **log you out immediately**.
  So we **cannot interactively log in** in the usual way.
    
## Solution(s)

### 1. Run a command directly via SSH
Instead of opening a shell, a command can be ran directly:
```bash
ssh bandit18@bandit.labs.overthewire.org -p 2220 cat readme
```
 - This logs in, executes `cat readme` **without running `.bashrc` interactively**, and prints the password.
    

 ## Explanation
When a command is passed to SSH, it doesn’t start an interactive shell. .bashrc is only sourced for interactive shells.
    
    
### 2. Use `scp` to copy the file locally
  
```bash
scp -P 2220 bandit18@bandit.labs.overthewire.org:readme .
cat readme
```
- This copies the file to the local machine and can be then read safely.
    
---
    
### 3. Temporary safe shell via `ssh -t`
    
If a shell is needed for more exploration, `.bashrc` can be bypassed like this: 
```bash
ssh bandit18@bandit.labs.overthewire.org -p 2220 'bash --noprofile --norc'
``` 
- `-norc` prevents `.bashrc` from running.
    
---
    
### Summary
  - `.bashrc` logs us out on this level.
  - **Solution:** don’t start an interactive shell. Run the command directly: `cat readme` or `scp` it out.
  - The **readme** file contains the **password for bandit19**.

# Level 19

## Challenge
To gain access to the next level, you should use the setuid binary in the homedirectory. Execute it without arguments to find out how to use it. The password for this level can be found in the usual place (/etc/bandit_pass), after you have used the setuid binary.
    
## Problem
There a **setuid binary** in the home directory
- This binary runs **as user bandit20**, even though we are logged in as **bandit19**.
- It must be used to read the file:
   ```
   /etc/bandit_pass/bandit20
   ```
 
  which normally **cannot** be read.
    
    ---
    
## Solution
1. Check the files:
 ```bash
 ls -l
 ```
Output :
    
```
total 16
-rwsr-x--- 1 bandit20 bandit19 14884 Oct 14 09:26 bandit20-do
```
    
The important part:
    
- `s` ⇒ **setuid bit**, meaning:
 - *When it's ran, it runs with bandit20’s permissions.*
     
2. Use the setuid binary to read the password
    
The trick is to give it a command that reads the password file.
   Example:
    ```bash
    ./bandit20-do cat /etc/bandit_pass/bandit20
    ```
Because the binary runs **as bandit20**, the `cat` command has permission to read the file. This switches the identity to bandit20.

# Level 20

## Challenge
There is a setuid binary in the homedirectory that does the following: it makes a connection to localhost on the port you specify as a commandline argument. It then reads a line of text from the connection and compares it to the password in the previous level (bandit20). If the password is correct, it will transmit the password for the next level (bandit21).
    
## Problem - **What the setuid binary does**
    
Inside the home directory:
    
```
-rwsr-x--- 1 bandit21 bandit20 14876 ... suconnect
```
    
This binary runs as **bandit21** because of the `s` (setuid).
    
If:
 ```
 ./suconnect <port>
 ```
    
it:
    
1. connects to `localhost:<port>`
2. waits for *us* to send **one line of text**
3. compares the line to the bandit20 password
4. **if correct**, it sends the bandit21 password through that same connection

This means:
    
- **A listening port** must be created (like a server)
- The binary will **connect to us**
- We **send it the bandit20 password**
- It will **respond with the bandit21 password**
    
    ---
    
## Solution
    
**Step 1 — Pick an unused port**
    
Pick a port > 1024 (only root can use ports < 1024).
    
 
**Step 2 — Start a listener**
```
nc -lvp 12345
```  

This means:
  - **l** = listen
  - **v** = verbose
  - **p** = port number
    
Leave this running — it waits for connections.
    
---
    
**Step 3 — In another terminal (or a second login), run the setuid binary**
Open a second terminal and SSH into bandit20 again.
```
./suconnect 12345
```

This will:
  - connect to the listener
  - expect the **bandit20 password** to be sent
    
---
    
**Step 4 — Send the bandit20 password**
    
  In the **first terminal** (the listener),
  paste the bandit20 password and press **Enter**.
 
    
## Summary
1. **Terminal 1**
 ```
 nc -lvp 12345
 ```
        
2. **Terminal 2**
        
  ```
  ./suconnect 12345
  ```
        
3. **Terminal 1**
        
    Paste the **bandit20 password**
        
4. **Terminal 1**
        
   Receive the **bandit21 password**
